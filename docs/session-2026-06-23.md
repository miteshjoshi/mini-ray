# Session Notes: 2026-06-23

## Focus

This session moved `mini-ray` from basic task execution toward a more Ray-like actor execution model with worker placement.

## Implemented

### Head Server Binary

Added a runnable `mini-ray-head` binary:

```bash
cargo run -p mini-ray-head -- --bind 127.0.0.1:50051
```

The binary starts a tonic gRPC server and serves `HeadService::with_defaults()`.

### Worker Runtime Loop

Implemented the worker execution path:

```text
connect to head
register worker
heartbeat
poll task leases
fetch dependency objects
execute via runtime registry
complete or fail task
```

The worker binary now supports:

```bash
cargo run -p mini-ray-worker -- --head http://127.0.0.1:50051 --slots 1
```

### Task Runtime Registry

Added `TaskRegistry` in `mini-ray-runtime`.

It supports:

```rust
register_raw(...)
register_nullary(...)
register_unary(...)
register_binary(...)
execute(...)
```

The worker CLI registers demo task functions:

```text
add_one(u64) -> u64
add(u64, u64) -> u64
```

### Actor Runtime Foundation

Added actor-side runtime pieces:

```text
ActorRegistry
ActorInstanceStore
actor constructors
actor methods
stateful actor instances
```

The demo actor registered by the worker is:

```text
Counter
increment(delta: u64) -> u64
```

### Actor Worker Placement

Implemented basic actor pinning:

```text
first worker to poll an actor task becomes the actor owner
future actor method tasks are pinned to that worker
non-owner workers cannot poll pinned actor tasks
pinned actor tasks are not stolen by other workers
```

This aligns with the Ray idea that actor methods must run where actor state lives.

## Important Design Notes

The object store is still required even with actors.

```text
object store = shared immutable input/output bytes
actor = mutable stateful execution entity
```

Actor constructor args, method args, and method results still flow through object refs and the object store.
